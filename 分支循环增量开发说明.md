# 分支循环增量开发说明

## 1. 这份文档解决什么问题

这份文档面向“已经有表达式、赋值、语句块、变量声明，但还没有完整控制流”的 MiniC 编译器前端。

目标不是一次性重构整个编译器，而是按照现有工程结构，**增量**加入下面这些能力：

1. `if`
2. `if ... else`
3. `while`
4. `break`
5. `continue`
6. 条件表达式里需要的比较与逻辑运算

也就是说，这次不是只把 `if` 和 `while` 这两个关键字塞进语法里，而是要把一条完整链路打通：

`文法 -> AST -> CST Visitor -> IREmitter -> 本地测试`

---

## 2. 为什么分支循环不能只改一个文件

在这个项目里，前端和 IR 的关系是非常直接的：

1. `frontend/antlr4/MiniC.g4`
   负责定义“语言能写什么”
2. `frontend/antlr4/Antlr4CSTVisitor.cpp`
   负责把 ANTLR 的 CST 转成项目自己的 AST
3. `frontend/AST.h`
   负责定义 AST 节点种类
4. `ir/IREmitter.cpp`
   负责把 AST 翻译成 LLVM IR 文本

所以分支循环的增量开发，至少要回答四个问题：

1. 文法怎么识别 `if` / `while`？
2. AST 用什么节点表示 `if` / `while` / `break` / `continue`？
3. Visitor 怎么把语法树变成这些 AST 节点？
4. IREmitter 怎么把这些 AST 节点翻译成基本块和跳转？

如果只改文法，不改 AST 和 IR，结果通常是：

- 语法能过，但生成 AST 失败
- 或 AST 能生成，但 `IREmitter` 找不到对应节点类型
- 或 IR 能生成一部分，但控制流结构不对

---

## 3. 增量开发前先明确边界

这次增量实现里，建议先把范围收紧，不要一口气做太多语义特性。

### 3.1 先做的内容

1. `if (expr) stmt`
2. `if (expr) stmt else stmt`
3. `while (expr) stmt`
4. `break;`
5. `continue;`
6. 条件表达式中的：
   - `< > <= >=`
   - `== !=`
   - `&& || !`

### 3.2 暂时不要顺手做太多

下面这些经常和控制流一起出现，但它们属于另一批增量：

1. 函数形参
2. 数组
3. 多维数组
4. 短路逻辑的更精细语义优化
5. `for`
6. `do while`
7. `switch`

原因很简单：如果你同时改太多维度，出现 bug 时很难判断到底是控制流错了，还是数组/函数参数本身还没支持。

## 4. 第一步：扩展文法

对应文件：

- `frontend/antlr4/MiniC.g4`

### 4.1 给 `statement` 增加控制流产生式

原来 `statement` 只有：

1. `return`
2. 赋值
3. 语句块
4. 表达式语句

现在需要增加：

```antlr
statement:
    T_RETURN expr T_SEMICOLON                        # returnStatement
    | lVal T_ASSIGN expr T_SEMICOLON                # assignStatement
    | T_IF T_L_PAREN expr T_R_PAREN statement (T_ELSE statement)? # ifStatement
    | T_WHILE T_L_PAREN expr T_R_PAREN statement    # whileStatement
    | T_BREAK T_SEMICOLON                           # breakStatement
    | T_CONTINUE T_SEMICOLON                        # continueStatement
    | block                                         # blockStatement
    | expr? T_SEMICOLON                             # expressionStatement
    ;
```

这次实际在 `frontend/antlr4/MiniC.g4` 中加入的核心代码如下：

```antlr
statement:
	T_RETURN expr T_SEMICOLON			# returnStatement
	| lVal T_ASSIGN expr T_SEMICOLON	# assignStatement
	| T_IF T_L_PAREN expr T_R_PAREN statement (T_ELSE statement)? # ifStatement
	| T_WHILE T_L_PAREN expr T_R_PAREN statement # whileStatement
	| T_BREAK T_SEMICOLON				# breakStatement
	| T_CONTINUE T_SEMICOLON			# continueStatement
	| block								# blockStatement
	| expr? T_SEMICOLON					# expressionStatement;
```

### 4.2 为什么 `if` / `while` 用的是 `statement` 而不是 `block`

因为我们希望同时支持：

```c
if (a) b = 1;
while (x) x = x - 1;
```

也支持：

```c
if (a) {
  b = 1;
}
```

所以分支和循环体的文法对象应该是 `statement`。

### 4.3 条件表达式不能继续只用加减乘除

如果没有比较和逻辑表达式，那么像下面这些都没法写：

```c
if (a < b)
while (x != 0)
if (a == 1 && b < 3)
```

所以需要把 `expr` 扩成一套带优先级的表达式体系，例如：

```antlr
expr: lOrExp;
lOrExp: lAndExp (T_LOR lAndExp)*;
lAndExp: eqExp (T_LAND eqExp)*;
eqExp: relExp (eqOp relExp)*;
relExp: addExp (relOp addExp)*;
```

这次实际加入的表达式层次是：

```antlr
expr: lOrExp;

lOrExp: lAndExp (T_LOR lAndExp)*;
lAndExp: eqExp (T_LAND eqExp)*;
eqExp: relExp (eqOp relExp)*;
eqOp: T_EQ | T_NE;
relExp: addExp (relOp addExp)*;
relOp: T_LT | T_GT | T_LE | T_GE;

addExp: mulExp (addOp mulExp)*;
mulExp: unaryExp (mulOp unaryExp)*;
unaryExp: primaryExp | T_ID T_L_PAREN realParamList? T_R_PAREN | unaryOp unaryExp;
unaryOp: T_ADD | T_SUB | T_NOT;
```

这样优先级关系就是：

1. `||`
2. `&&`
3. `== !=`
4. `< > <= >=`
5. `+ -`
6. `* / %`
7. 单目 `+ - !`

### 4.4 记得补词法 token

至少要补：

```antlr
T_IF: 'if';
T_ELSE: 'else';
T_WHILE: 'while';
T_BREAK: 'break';
T_CONTINUE: 'continue';

T_EQ: '==';
T_NE: '!=';
T_LE: '<=';
T_GE: '>=';
T_LT: '<';
T_GT: '>';
T_NOT: '!';
T_LAND: '&&';
T_LOR: '||';
```

实际补到 `MiniC.g4` 中的关键 token 片段如下：

```antlr
T_COMMA: ',';
T_ASSIGN: '=';

T_EQ: '==';
T_NE: '!=';
T_LE: '<=';
T_GE: '>=';
T_LT: '<';
T_GT: '>';

T_ADD: '+';
T_SUB: '-';
T_MUL: '*';
T_DIV: '/';
T_MOD: '%';
T_NOT: '!';
T_LAND: '&&';
T_LOR: '||';

T_IF: 'if';
T_ELSE: 'else';
T_WHILE: 'while';
T_BREAK: 'break';
T_CONTINUE: 'continue';
```

这里有两个常见坑：

1. `==` 必须放在 `=` 的规则逻辑之前考虑
2. `<= >= !=` 这类双字符 token 必须先于单字符版本匹配

ANTLR 词法按最长匹配原则通常能帮你挡掉一部分问题，但写规则时最好仍然有这个意识。

## 5. 第二步：扩展 AST 节点类型

对应文件：

- `frontend/AST.h`

### 5.1 为什么要先改 AST 枚举

`IREmitter` 并不是直接看 ANTLR 的语法上下文，它只看项目自己的 `ast_operator_type`。

所以只要要新增一种语义结构，就必须先给它一个 AST 节点类型。

### 5.2 这次新增的节点建议

控制流节点：

1. `AST_OP_IF`
2. `AST_OP_WHILE`
3. `AST_OP_BREAK`
4. `AST_OP_CONTINUE`

条件表达式节点：

1. `AST_OP_LT`
2. `AST_OP_GT`
3. `AST_OP_LE`
4. `AST_OP_GE`
5. `AST_OP_EQ`
6. `AST_OP_NE`
7. `AST_OP_LAND`
8. `AST_OP_LOR`
9. `AST_OP_NOT`

这次在 `frontend/AST.h` 中新增的核心枚举代码如下：

```cpp
/// @brief if/if-else 语句运算符
AST_OP_IF,

/// @brief while 语句运算符
AST_OP_WHILE,

/// @brief break 语句运算符
AST_OP_BREAK,

/// @brief continue 语句运算符
AST_OP_CONTINUE,

/// @brief 二元关系运算符<
AST_OP_LT,
/// @brief 二元关系运算符>
AST_OP_GT,
/// @brief 二元关系运算符<=
AST_OP_LE,
/// @brief 二元关系运算符>=
AST_OP_GE,
/// @brief 二元相等运算符==
AST_OP_EQ,
/// @brief 二元不等运算符!=
AST_OP_NE,
/// @brief 逻辑与运算符&&
AST_OP_LAND,
/// @brief 逻辑或运算符||
AST_OP_LOR,
/// @brief 单目运算符! (逻辑非)
AST_OP_NOT,
```

### 5.3 建议明确每类节点的孩子约定

这是最重要的点之一，因为 `IREmitter` 会按位置取孩子。

建议约定如下：

#### `AST_OP_IF`

孩子顺序：

1. 条件表达式
2. then 分支
3. else 分支，可选

#### `AST_OP_WHILE`

孩子顺序：

1. 条件表达式
2. 循环体

#### `AST_OP_ASSIGN`

孩子顺序仍然保持：

1. 左值
2. 右值

#### 二元比较 / 逻辑运算

孩子顺序仍然保持：

1. 左操作数
2. 右操作数

这一步一定要在文档和代码里都保持一致，不然后面 `IREmitter` 很容易因为下标假设写错。

## 6. 第三步：在 Visitor 中把 CST 变成 AST

对应文件：

- `frontend/antlr4/Antlr4CSTVisitor.h`
- `frontend/antlr4/Antlr4CSTVisitor.cpp`

这一步本质上是在做“语法到语义树”的归一化。

### 6.1 先在 `.h` 里补新的 visit 函数声明

典型包括：

1. `visitIfStatement`
2. `visitWhileStatement`
3. `visitBreakStatement`
4. `visitContinueStatement`
5. `visitLOrExp`
6. `visitLAndExp`
7. `visitEqExp`
8. `visitEqOp`
9. `visitRelExp`
10. `visitRelOp`

这次在 `frontend/antlr4/Antlr4CSTVisitor.h` 里加入的声明如下：

```cpp
std::any visitIfStatement(MiniCParser::IfStatementContext * ctx) override;
std::any visitWhileStatement(MiniCParser::WhileStatementContext * ctx) override;
std::any visitBreakStatement(MiniCParser::BreakStatementContext * ctx) override;
std::any visitContinueStatement(MiniCParser::ContinueStatementContext * ctx) override;

std::any visitLOrExp(MiniCParser::LOrExpContext * ctx) override;
std::any visitLAndExp(MiniCParser::LAndExpContext * ctx) override;
std::any visitEqExp(MiniCParser::EqExpContext * ctx) override;
std::any visitEqOp(MiniCParser::EqOpContext * ctx) override;
std::any visitRelExp(MiniCParser::RelExpContext * ctx) override;
std::any visitRelOp(MiniCParser::RelOpContext * ctx) override;
```

### 6.2 在 `visitStatement` 里接入口

原来这里只分发：

1. 赋值
2. return
3. block
4. expression

现在要把 `if / while / break / continue` 也接上。

这一步如果漏了，文法虽然已经能识别，但访问到对应语法树时会直接返回空。

实际分发代码如下：

```cpp
if (Instanceof(assignCtx, MiniCParser::AssignStatementContext *, ctx)) {
	return visitAssignStatement(assignCtx);
} else if (Instanceof(returnCtx, MiniCParser::ReturnStatementContext *, ctx)) {
	return visitReturnStatement(returnCtx);
} else if (Instanceof(ifCtx, MiniCParser::IfStatementContext *, ctx)) {
	return visitIfStatement(ifCtx);
} else if (Instanceof(whileCtx, MiniCParser::WhileStatementContext *, ctx)) {
	return visitWhileStatement(whileCtx);
} else if (Instanceof(breakCtx, MiniCParser::BreakStatementContext *, ctx)) {
	return visitBreakStatement(breakCtx);
} else if (Instanceof(continueCtx, MiniCParser::ContinueStatementContext *, ctx)) {
	return visitContinueStatement(continueCtx);
}
```

### 6.3 `if` 节点怎么造

建议逻辑：

1. 访问条件表达式
2. 访问 then 语句
3. 如果有 else，就访问 else 语句
4. 组装成 `AST_OP_IF`

伪代码：

```cpp
condNode = visitExpr(...)
thenNode = visitStatement(...)
elseNode = visitStatement(...) // 可选
return AST_OP_IF(condNode, thenNode, elseNode)
```

实际实现代码如下：

```cpp
std::any MiniCCSTVisitor::visitIfStatement(MiniCParser::IfStatementContext * ctx)
{
	auto condNode = std::any_cast<ast_node *>(visitExpr(ctx->expr()));
	auto thenNode = ensureStatementNode(std::any_cast<ast_node *>(visitStatement(ctx->statement(0))));

	ast_node * elseNode = nullptr;
	if (ctx->statement().size() > 1) {
		elseNode = ensureStatementNode(std::any_cast<ast_node *>(visitStatement(ctx->statement(1))));
	}

	return ast_node::New(ast_operator_type::AST_OP_IF, condNode, thenNode, elseNode);
}
```

### 6.4 `while` 节点怎么造

逻辑更简单：

```cpp
condNode = visitExpr(...)
bodyNode = visitStatement(...)
return AST_OP_WHILE(condNode, bodyNode)
```

实际实现代码如下：

```cpp
std::any MiniCCSTVisitor::visitWhileStatement(MiniCParser::WhileStatementContext * ctx)
{
	auto condNode = std::any_cast<ast_node *>(visitExpr(ctx->expr()));
	auto bodyNode = ensureStatementNode(std::any_cast<ast_node *>(visitStatement(ctx->statement())));

	return ast_node::New(ast_operator_type::AST_OP_WHILE, condNode, bodyNode);
}
```

### 6.5 为什么要给空语句准备一个“兜底节点”

像下面这种写法是合法的：

```c
while (a < 10) ;
```

如果 `expressionStatement` 对空语句直接返回 `nullptr`，而循环体又没处理这个情况，后面 IR 阶段就可能访问空指针。

一个稳妥办法是：

1. 对真正的空语句仍然允许返回 `nullptr`
2. 但在 `if` / `while` 组装语句体时，统一用一个辅助函数把空语句包装成“空 block 节点”

这样 IR 阶段仍然面对的是一个正常 AST 节点。

这次的辅助代码如下：

```cpp
namespace {

ast_node * makeEmptyStmtNode()
{
	auto * node = ast_node::New(ast_operator_type::AST_OP_BLOCK);
	node->needScope = false;
	return node;
}

ast_node * ensureStatementNode(ast_node * node)
{
	return node ? node : makeEmptyStmtNode();
}

}  // namespace
```

### 6.6 比较和逻辑表达式怎么造

和加减乘除一样，推荐统一成“左结合折叠”。

例如：

```c
a < b < c
```

在 AST 中不是三个孩子的平铺节点，而是：

```text
      <
     / \
    <   c
   / \
  a   b
```

同理，`&&` 和 `||` 在 AST 层可以先当普通二元节点处理，短路语义留给 IR 阶段。

实际 Visitor 中的折叠代码可以参考：

```cpp
std::any MiniCCSTVisitor::visitRelExp(MiniCParser::RelExpContext * ctx)
{
	if (ctx->relOp().empty()) {
		return visitAddExp(ctx->addExp()[0]);
	}

	ast_node * left = std::any_cast<ast_node *>(visitAddExp(ctx->addExp()[0]));
	for (size_t i = 0; i < ctx->relOp().size(); i++) {
		auto op = std::any_cast<ast_operator_type>(visitRelOp(ctx->relOp()[i]));
		auto * right = std::any_cast<ast_node *>(visitAddExp(ctx->addExp()[i + 1]));
		left = ast_node::New(op, left, right);
	}

	return left;
}
```

### 6.7 单目 `!` 的处理

和单目 `-` 一样：

1. 先访问内部 `unaryExp`
2. 再包成 `AST_OP_NOT`

如果是单目 `+`，最简单的做法通常是**不额外造节点，直接返回操作数**。

这次 `unaryOp` 和 `unaryExp` 的实际处理如下：

```cpp
std::any MiniCCSTVisitor::visitUnaryOp(MiniCParser::UnaryOpContext * ctx)
{
	if (ctx->T_ADD()) {
		return ast_operator_type::AST_OP_ADD;
	} else if (ctx->T_SUB()) {
		return ast_operator_type::AST_OP_NEG;
	}

	return ast_operator_type::AST_OP_NOT;
}

std::any MiniCCSTVisitor::visitUnaryExp(MiniCParser::UnaryExpContext * ctx)
{
	...
	ast_node * operand = std::any_cast<ast_node *>(visitUnaryExp(ctx->unaryExp()));
	ast_operator_type op = std::any_cast<ast_operator_type>(visitUnaryOp(ctx->unaryOp()));

	if (op == ast_operator_type::AST_OP_ADD) {
		return operand;
	}

	return ast_node::New(op, operand);
}
```

## 7. 第四步：在 IREmitter 中接入新 AST 节点

对应文件：

- `ir/IREmitter.h`
- `ir/IREmitter.cpp`

这是控制流增量里最核心的一步。

### 7.1 先注册新的 handler

在构造函数里把新节点类型挂到 visitor 函数上，例如：

1. `AST_OP_IF -> visitIf`
2. `AST_OP_WHILE -> visitWhile`
3. `AST_OP_BREAK -> visitBreak`
4. `AST_OP_CONTINUE -> visitContinue`
5. `AST_OP_LT / GT / LE / GE / EQ / NE`
6. `AST_OP_LAND / LOR / NOT`

如果这一步漏掉，运行时会落到：

```cpp
Unknown AST node type
```

这次在 `ir/IREmitter.cpp` 构造函数里新增的注册代码如下：

```cpp
handlers[ast_operator_type::AST_OP_LT] = &IREmitter::visitLt;
handlers[ast_operator_type::AST_OP_GT] = &IREmitter::visitGt;
handlers[ast_operator_type::AST_OP_LE] = &IREmitter::visitLe;
handlers[ast_operator_type::AST_OP_GE] = &IREmitter::visitGe;
handlers[ast_operator_type::AST_OP_EQ] = &IREmitter::visitEq;
handlers[ast_operator_type::AST_OP_NE] = &IREmitter::visitNe;
handlers[ast_operator_type::AST_OP_LAND] = &IREmitter::visitLogicalAnd;
handlers[ast_operator_type::AST_OP_LOR] = &IREmitter::visitLogicalOr;
handlers[ast_operator_type::AST_OP_NOT] = &IREmitter::visitNot;

handlers[ast_operator_type::AST_OP_IF] = &IREmitter::visitIf;
handlers[ast_operator_type::AST_OP_WHILE] = &IREmitter::visitWhile;
handlers[ast_operator_type::AST_OP_BREAK] = &IREmitter::visitBreak;
handlers[ast_operator_type::AST_OP_CONTINUE] = &IREmitter::visitContinue;
```

### 7.2 为什么比较表达式最终要返回 `i32` 而不是 `i1`

这个项目现在的表达式体系整体上把“表达式值”当成 `i32` 处理。

所以比较表达式和逻辑表达式通常更适合走下面这条路：

1. 先发出 `icmp`，得到 `i1`
2. 再 `zext i1 -> i32`
3. 把结果当普通表达式值返回

这样可以兼容：

```c
int x;
x = (a < b);
return a == b;
```

这次对应的辅助函数实现如下：

```cpp
std::string IREmitter::emitCondValue(const IRValue & value)
{
    std::string reg = nextReg();
    emit(reg + " = icmp ne i32 " + value.text + ", 0");
    return reg;
}

std::string IREmitter::emitBoolToInt(const std::string & boolReg)
{
    std::string reg = nextReg();
    emit(reg + " = zext i1 " + boolReg + " to i32");
    return reg;
}
```

### 7.3 条件跳转前为什么还要做一次“条件归一化”

虽然表达式返回的是 `i32`，但 LLVM 的 `br i1` 需要的是真正的布尔值。

所以在 `if` / `while` 里，建议统一做：

```llvm
%cond = icmp ne i32 <expr>, 0
br i1 %cond, ...
```

也就是说：

- 表达式阶段统一返回 `i32`
- 控制流阶段再把它转成 `i1`

这样层次清晰，也不容易乱。

这就是为什么 `visitLt` / `visitEq` 这类函数最后不是直接返回 `i1`，而是：

```cpp
std::string boolReg = nextReg();
emit(boolReg + " = icmp slt i32 " + lhs.text + ", " + rhs.text);
return {emitBoolToInt(boolReg), false};
```

### 7.4 `if` 的 IR 结构怎么搭

#### 无 `else`

基本结构：

```text
当前块
  -> 条件判断
  -> br cond, then, end

then:
  ...
  br end

end:
```

#### 有 `else`

基本结构：

```text
当前块
  -> 条件判断
  -> br cond, then, else

then:
  ...
  br end

else:
  ...
  br end

end:
```

这里最容易犯的错是：

1. then/else 已经 `ret` 或 `br` 了，还重复补一条 `br end`
2. then 和 else 都终结了，但还强行创建一个无入口的 `end` 块

所以实现时通常要结合 `hasTerminator` 判断。

这次 `visitIf` 的实际代码如下：

```cpp
IREmitter::IRValue IREmitter::visitIf(ast_node * node)
{
    ast_node * condNode = node->sons[0];
    ast_node * thenNode = node->sons[1];
    ast_node * elseNode = node->sons.size() >= 3 ? node->sons[2] : nullptr;

    IRValue condVal = visitNode(condNode);
    std::string condReg = emitCondValue(condVal);
    std::string thenLabel = nextLabel();
    std::string endLabel = nextLabel();

    if (!elseNode) {
        emit("br i1 " + condReg + ", label %" + thenLabel + ", label %" + endLabel);
        hasTerminator = true;

        startBlock(thenLabel);
        IRValue thenResult = visitNode(thenNode);
        if (!hasTerminator) {
            emit("br label %" + endLabel);
            hasTerminator = true;
        }

        startBlock(endLabel);
        return {"__ok", false};
    }

    std::string elseLabel = nextLabel();
    emit("br i1 " + condReg + ", label %" + thenLabel + ", label %" + elseLabel);
    hasTerminator = true;

    startBlock(thenLabel);
    IRValue thenResult = visitNode(thenNode);
    bool thenTerminated = hasTerminator;
    if (!thenTerminated) {
        emit("br label %" + endLabel);
        hasTerminator = true;
    }

    startBlock(elseLabel);
    IRValue elseResult = visitNode(elseNode);
    bool elseTerminated = hasTerminator;
    if (!elseTerminated) {
        emit("br label %" + endLabel);
        hasTerminator = true;
    }

    if (thenTerminated && elseTerminated) {
        hasTerminator = true;
        return {"__ok", false};
    }

    startBlock(endLabel);
    return {"__ok", false};
}
```

### 7.5 `while` 的 IR 结构怎么搭

推荐结构：

```text
当前块:
  br cond

cond:
  %c = ...
  br i1 %c, body, end

body:
  ...
  br cond

end:
```

这个结构简单、稳定，而且和 `continue` 天然契合。

这次 `visitWhile` 的实际代码如下：

```cpp
IREmitter::IRValue IREmitter::visitWhile(ast_node * node)
{
    ast_node * condNode = node->sons[0];
    ast_node * bodyNode = node->sons[1];

    std::string condLabel = nextLabel();
    std::string bodyLabel = nextLabel();
    std::string endLabel = nextLabel();

    emit("br label %" + condLabel);
    hasTerminator = true;

    startBlock(condLabel);
    IRValue condVal = visitNode(condNode);
    std::string condReg = emitCondValue(condVal);
    emit("br i1 " + condReg + ", label %" + bodyLabel + ", label %" + endLabel);
    hasTerminator = true;

    breakLabels.push_back(endLabel);
    continueLabels.push_back(condLabel);

    startBlock(bodyLabel);
    IRValue bodyResult = visitNode(bodyNode);
    if (!hasTerminator) {
        emit("br label %" + condLabel);
        hasTerminator = true;
    }

    breakLabels.pop_back();
    continueLabels.pop_back();

    startBlock(endLabel);
    return {"__ok", false};
}
```

### 7.6 `break` / `continue` 为什么需要标签栈

一层循环时，`break` 和 `continue` 很简单：

1. `break -> endLabel`
2. `continue -> condLabel`

但嵌套循环时：

```c
while (...) {
  while (...) {
    break;
  }
}
```

内层 `break` 只能跳到内层结束标签，不能跳到外层。

所以最稳妥的做法是维护两个栈：

1. `breakLabels`
2. `continueLabels`

进入一个循环时压栈，离开时弹栈。

这样：

- `break` 永远跳 `breakLabels.back()`
- `continue` 永远跳 `continueLabels.back()`

这次对应实现如下：

```cpp
IREmitter::IRValue IREmitter::visitBreak(ast_node * node)
{
    (void) node;

    if (breakLabels.empty()) {
        minic_log(LOG_ERROR, "break 语句不在循环内");
        return {};
    }

    emit("br label %" + breakLabels.back());
    hasTerminator = true;
    return {"__ok", false};
}

IREmitter::IRValue IREmitter::visitContinue(ast_node * node)
{
    (void) node;

    if (continueLabels.empty()) {
        minic_log(LOG_ERROR, "continue 语句不在循环内");
        return {};
    }

    emit("br label %" + continueLabels.back());
    hasTerminator = true;
    return {"__ok", false};
}
```

### 7.7 逻辑与 / 逻辑或是否必须做短路

严格说，语义上应该做短路。

但增量开发时可以分两步：

#### 第一步：先做“结果正确”的实现

即便实现方式比较朴素，只要：

1. `&&` 最终返回 0/1
2. `||` 最终返回 0/1
3. 分支条件能正确工作

就能先把控制流打通。

#### 第二步：再优化为更干净的短路 CFG

也就是进一步减少临时 `alloca`、合并基本块、让逻辑表达式的控制流更像正式编译器。

对于教学和增量开发来说，**先保证功能通，再逐步优化** 是更稳的做法。

这次 `&&` / `||` 的实现就是这种“先保证结果正确”的版本。它们借助临时 `alloca` 保存结果，再根据左操作数决定是否进入右侧分支。例如：

```cpp
IREmitter::IRValue IREmitter::visitLogicalAnd(ast_node * node)
{
    IRValue lhs = visitNode(node->sons[0]);

    std::string resultPtr = nextReg();
    emit(resultPtr + " = alloca i32");
    emit("store i32 0, i32* " + resultPtr);

    std::string rhsLabel = nextLabel();
    std::string endLabel = nextLabel();

    std::string lhsCond = emitCondValue(lhs);
    emit("br i1 " + lhsCond + ", label %" + rhsLabel + ", label %" + endLabel);
    hasTerminator = true;

    startBlock(rhsLabel);
    IRValue rhs = visitNode(node->sons[1]);
    std::string rhsCond = emitCondValue(rhs);
    std::string rhsInt = emitBoolToInt(rhsCond);
    emit("store i32 " + rhsInt + ", i32* " + resultPtr);
    emit("br label %" + endLabel);
    hasTerminator = true;

    startBlock(endLabel);
    std::string reg = nextReg();
    emit(reg + " = load i32, i32* " + resultPtr);
    return {reg, false};
}
```

## 10. 这次增量里最容易踩的坑

### 10.1 只改了文法，没改 Visitor

表现：

1. ANTLR 解析通过
2. 但 AST 为空或部分节点是空
3. 后续 IR 访问空指针

### 10.2 AST 节点孩子顺序和 IREmitter 假设不一致

表现：

1. 不一定编译报错
2. 但运行结果完全不对
3. 常见于 `if`、`while`、变量声明、函数定义

### 10.3 空语句没有妥善处理

表现：

```c
while (a) ;
```

会在 IR 阶段访问空节点或生成不完整 CFG。

### 10.4 `break` / `continue` 没有栈

表现：

1. 单层循环测试通过
2. 嵌套循环测试跳错位置

### 10.5 比较表达式返回了 `i1`，普通表达式又假设是 `i32`

表现：

1. 某些 `if` 能过
2. 但 `x = (a < b)`、`return a == b` 一类表达式会出问题

### 10.6 忘记重新生成 ANTLR 文件

表现：

1. `.g4` 已经改了
2. 但 `MiniCParser.*` 还是旧版本
3. IDE 和编译器看到的是旧接口

## 11. 推荐的增量开发顺序

如果你后面还想继续往前推功能，推荐按下面顺序走：

1. `if`
2. `if else`
3. 比较运算 `< > <= >= == !=`
4. `while`
5. `break`
6. `continue`
7. `&& || !`
8. 函数形参
9. 数组

原因是：

1. `if/while` 能先把控制流框架搭起来
2. 比较运算是条件控制的基础
3. `break/continue` 只是在循环框架上加标签管理
4. 逻辑表达式会进一步增加控制流复杂度
5. 函数形参和数组是另一条复杂链路，最好和控制流分开推进

---

## 12. 一句话总结这次增量思路

分支与循环的增量开发，不是“往语法里加两个关键字”，而是：

1. 在文法里让它能被识别
2. 在 AST 里给它稳定的结构
3. 在 Visitor 里把 CST 正确归一化成 AST
4. 在 IREmitter 里把它翻译成基本块和跳转
5. 用分层测试把每一步都压实

如果按这条链路做，功能会长得比较稳，也更容易继续往 `函数形参`、`数组`、`更复杂控制流` 上扩展。
