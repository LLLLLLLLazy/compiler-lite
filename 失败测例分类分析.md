## 1. 分析范围

本文分析的对象是 [未通过测例情况.md](./未通过测例情况.md) 中列出的全部失败项，共 42 条失败记录。

## 2. 结论总览

按当前代码的实际失败原因，可以分成三大类。

| 类别 | 失败记录数 | 主要根因 |
| --- | ---: | --- |
| 功能未实现 | 26 | 全局数组初始化只支持空初始化列表 `{}`，不支持显式初始化值，也不支持 `{0}` 这种零初始化写法 |
| 功能实现错误有 bug | 12 | 函数预声明阶段无法解析全局 `const` 数组维度；超长表达式递归过深导致栈溢出；RISC-V 后端临时寄存器耗尽；RISC-V 浮点代码结果错误 |
| 功能实现但是性能差导致超时 | 4 | 超大 AST 图导出过慢；超深多分支程序编译时间过长 |

从优先级看，最值得先修的是“全局数组初始化支持不完整”，因为这一项单独覆盖了 26 条失败记录。

## 3. 功能未实现

### 3.1 全局数组初始化只支持 `{}`，不支持非空初始化列表

直接证据来自 `frontend/lowering/IRGenerator.cpp` 中的全局变量处理逻辑。

- `IRGenerator::visitVarDecl` 在处理全局数组时，遇到非空初始化列表会直接报错
- 关键位置在 `frontend/lowering/IRGenerator.cpp:643`
- 代码逻辑是：如果 `declType->isArrayType()`，则只有 `AST_OP_INIT_LIST && initNode->sons.empty()` 才被接受

这意味着当前实现只接受下面这一类全局数组初始化：

```c
int a[4] = {};
```

而下面这些都会直接失败：

```c
int a[4] = {1, 2, 3, 4};
int a[4] = {0};
const int a[4] = {1, 2, 3, 4};
int a[2][3] = {{1, 2, 3}, {4, 5, 6}};
```

因此有些测例虽然主题分别是“作用域遮蔽”“混乱 token”“长函数”“搜索”，但它们都会在更早阶段被同一个全局数组初始化限制拦住，根本还没走到测例真正想考的语义点。

## 4. 功能实现错误有 bug

### 4.1 函数预声明阶段过早构造形参类型，导致看不到前面的全局 `const` 维度

这组问题的直接现象是报错：`数组维度必须是正整数常量`。

根因不在“常量表达式求值能力完全缺失”，而在调用顺序。

- `IRGenerator::run` 先执行 `declareCompileUnit`，位置在 `frontend/lowering/IRGenerator.cpp:112-132`
- `declareCompileUnit` 会在真正遍历全局声明之前，先为所有函数调用 `buildDeclaredType(paramNode, true)` 构造形参类型，位置在 `frontend/lowering/IRGenerator.cpp:138-161`
- 但全局 `const` 绑定只有在后续 `visitCompileUnit -> visitDeclStmt -> visitVarDecl` 时，才会写入 `constBindings.front()`，位置在 `frontend/lowering/IRGenerator.cpp:560-685`
- `buildDeclaredType` 又要求所有数组维度立即能通过 `evaluateConstIntExpr` 求出来，位置在 `frontend/lowering/IRGenerator.cpp:680-709`

所以像下面这种函数形参：

```c
const int V = 4;
int isSafe(int graph[][V], int color[])
```

虽然从源码顺序看 `V` 已经定义在前面，但函数预声明时 `V` 其实还没进入 `constBindings`，于是会被误判成“不是正整数常量”。

#### 受影响失败项

2025 LLVM IR:

- `2025_func_015_graph_coloring`
- `2025_func_037_dct`

2025 RISCV64 asm:

- `2025_func_015_graph_coloring`
- `2025_func_037_dct`

#### 判断依据

- `2025_func_015_graph_coloring` 使用 `int graph[][V]`
- `2025_func_037_dct` 使用 `float mat[][MAX_DIM_Y]`
- 两者都在编译早期报同一个错误：`IRGenerator.cpp:703 数组维度必须是正整数常量`

### 4.2 超长左结合表达式导致 `visitExpr` / `emitBinary` 递归过深，最终栈溢出

这组问题对应 `long_code2`。

- `emitBinary` 递归地对左右子树再次调用 `visitExpr`，位置在 `frontend/lowering/IRGenerator.cpp:1415-1416`
- `long_code2` 的主体是极长的加法链，AST 高度非常大
- 在 `-L` / `-I` 编译时，AddressSanitizer 栈回溯反复出现在 `visitExpr -> emitBinary -> visitExpr -> emitBinary` 上，最终报 `stack-overflow`

这不是“该功能不存在”，而是现有递归实现对极长表达式不够稳健。

#### 受影响失败项

2023 LLVM IR:

- `2023_func_86_long_code2`

2023 RISCV64 asm:

- `2023_func_86_long_code2`

2025 LLVM IR:

- `2025_func_086_long_code2`

2025 RISCV64 asm:

- `2025_func_086_long_code2`

#### 额外说明

- 这个问题和 AST 模式下的 `long_code2` 失败不是同一原因
- LLVM IR / asm 模式下是 IR 生成阶段递归过深导致崩溃
- AST 模式下则是图导出过慢，属于性能问题，见第 5 节

### 4.3 RISC-V 后端临时寄存器池耗尽，直接 `abort`

这组问题只出现在 asm 模式，LLVM IR 模式能够通过，因此可以确定是后端 bug。

- 错误信息由 `backend/riscv64/LocalTempManager.cpp:136` 打印：`LocalTempManager: 无可用的临时寄存器！`
- 同一函数紧接着调用 `std::abort()`，位置在 `backend/riscv64/LocalTempManager.cpp:136-137`

这表明指令选择或地址计算阶段同时借用了过多局部临时寄存器，而当前释放策略没有兜住高复杂度场景。

#### 受影响失败项

2023 RISCV64 asm:

- `2023_func_94_nested_loops`

2025 RISCV64 asm:

- `2025_func_030_many_dimensions`
- `2025_func_094_nested_loops`

#### 判断依据

- `2025_func_030_many_dimensions` 的 LLVM IR 可以成功生成，但 asm 编译直接 abort，说明是后端问题
- `2023_func_94_nested_loops` / `2025_func_094_nested_loops` 都是高维数组访问和深层循环，容易在地址展开时堆高临时值数量

### 4.4 RISC-V 浮点后端结果错误，程序能跑完但输出和标答不一致

这组问题目前只定位到“RISC-V 后端浮点相关 bug”，还没有继续下钻到某一条具体指令选择规则。

#### 受影响失败项

2025 RISCV64 asm:

- `2025_func_038_light2d`

#### 判断依据

- 该测例不在 LLVM IR 失败列表中，只在 asm 模式失败，说明前端和 IR 语义大体是通的，问题落在后端
- 该程序可正常编译、链接、运行并在 40 秒内结束，因此不是超时
- 但把实际输出与标答 diff 后，第一屏就出现大量像素值不一致。例如标答前两行是：

```text
0 31 63 31 31 63 63 31 31 63 31 31
31 31 31 63 31 31 63 31 31 63 31 31
```

而当前 asm 输出前两行变成了：

```text
31 31 31 63 31 63 31 31 0 0 63 31
31 31 31 63 63 63 63 63 31 0 31 0
```

结合源码特征看，最可疑的是 RISC-V 浮点算术、浮点比较、浮点调用约定、或浮点与整数互转中的某一处实现有误。

## 5. 功能已实现，但性能差导致超时

### 5.1 AST 图导出对超大 AST 不友好

AST 模式失败的两个用例，本质上都不是前端无法解析，而是 AST 图导出太重。

#### 受影响失败项

2023 AST:

- `2023_func_83_long_array`
- `2023_func_86_long_code2`

#### 判断依据

- `2023_func_83_long_array` 在 AST 导出时出现 Graphviz / cairo 提示：`graph is too large for cairo-renderer bitmaps`，随后超时
- `2023_func_86_long_code2` AST 模式下也在 30 秒内无法完成
- 这说明 AST 可视化链路对超大表达式树或超大语法树的缩放、布局、位图渲染开销过大

### 5.2 深层多分支程序的编译时间过长

`multi_branch` 当前最符合“实现了，但编译性能差导致超时”的定义。

#### 受影响失败项

2025 LLVM IR:

- `2025_func_033_multi_branch`

2025 RISCV64 asm:

- `2025_func_033_multi_branch`

#### 判断依据

- `-L` 和 `-I` 编译都在 20 秒量级内无法结束
- AST 输出模式也无法在 25 秒内完成
- 源码本身是极深的 if-else 链，容易让语法树遍历、CFG 构造、以及后续打印都呈现很差的扩展性

这里我没有继续把热点细分到某一个函数级别，但从现象上看，它更像是编译时间扩展性问题，而不是单点逻辑错误。

## 7. 建议修复顺序

如果目的是尽快拉高通过率，建议按下面顺序处理。

1. 补全全局数组初始化支持

   这一步能一次性消掉 26 条失败记录，收益最高

2. 修复“函数预声明阶段无法解析全局 `const` 维度”

   这能解决 `graph_coloring` 和 `dct`

3. 修复超长表达式的递归栈溢出

   这能解决 `long_code2` 在 LLVM IR 和 asm 下的失败

4. 修 RISC-V 后端的临时寄存器耗尽

   这能解决 `many_dimensions` 和 `nested_loops` 这类 backend-only 失败

5. 单独排查 `light2d` 的浮点后端结果错误

   这是独立的 backend 浮点正确性问题

6. 最后再做 AST 导出和深层多分支的性能优化

   这些不会影响语义正确性，但会拖慢评测和工具链体验
